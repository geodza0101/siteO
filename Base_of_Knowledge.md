## 🗺️ МЕТА-КАРТА: Как читать эту базу

```
┌─────────────────────────────────────────────────────────────────────┐
│                        EBITDA / Чистая прибыль                       │
│                              ▲                                       │
│         ┌────────────────────┴────────────────────┐                  │
│         │                                         │                  │
│   REVENUE (Выручка)                    COSTS (Затраты)               │
│         │                                         │                  │
│    Buyers × LTV                      Fixed + Variable                │
│         │                                         │                  │
│   ┌─────┴─────┐                           ┌──────┴──────┐            │
│   │           │                           │             │            │
│ Conv(C1)   AMPPU                       CAC          COGS            │
│   │           │                           │             │            │
│   │      Av.Price × Margin           CPUser × Conv    Servers, API   │
│   │                                       │                          │
│   └───────────────────┬───────────────────┘                          │
│                       │                                              │
│              PRODUCT (Retention, Activation)                         │
│                       │                                              │
│              USERS (Acquisition)                                     │
│                       │                                              │
│              CHANNELS (Impressions → Clicks → CTR)                   │
└─────────────────────────────────────────────────────────────────────┘
```

**Принцип чтения:** Каждое изменение на нижнем уровне (например, CTR баннера) по цепочке бьёт в верхний уровень (EBITDA). Продакт должен видеть **всю цепь** и понимать, какой **рычаг** даёт максимальный эффект при минимальных усилиях.

# ЧАСТЬ I: ФУНДАМЕНТ МЫШЛЕНИЯ

## 1. Философия Продукта

### 1.1 Продукт vs Проект
| Характеристика | Проектный бизнес (Агентство) | Продуктовый бизнес (SaaS) |
|----------------|------------------------------|---------------------------|
| **Рост** | Линейный (x2 людей = x2 денег) | Экспоненциальный ("Клюшка") |
| **Расходы** | Растут с доходами | Fixed Costs почти не растут |
| **Масштаб** | Ограничен людьми | Неограничен (отвязка от человеко-часов) |
| **Риск** | Низкий, предсказуемый | Высокий на старте, потом стабильность |

### 1.2 Сердце любой продажи
```
Value (Ценность для клиента) > Price (Цена продукта)
```
*   Если конверсия низкая — либо цена завышена, либо **ценность не донесена**.
*   Клиент всегда взвешивает: "Стоит ли это моих денег?"

### 1.3 Универсальная бизнес-модель
Все бизнесы — одинаковы:
```
Channels → User Acquisition → Activation → Sales → Buyer → Payments → Gross Profit
                    ↓                              ↓
            Virality Loop                   Retention Loop
```

### 1.4 Типы монетизации
| Тип | Чек | Стратегия | LTV модель |
|-----|-----|-----------|------------|
| **Low-ticket** | 1-5K₽ | Игра в объём, тысячи лидов | Подписка, микротранзакции |
| **Mid-ticket** | 10-50K₽ | Автоматизация + касания | Freemium → Premium |
| **High-ticket** | 100K₽+ | Ручные продажи, "Киты" | Enterprise, контракты |

**Принцип:** Когда продукт работает — масштабируй вход (трафик), а не допиливай фичи.

## 2. Когнитивные ловушки (Biases)

### 2.1 Главные враги продакта
| Ловушка | Описание | Антидот |
|---------|----------|---------|
| **Ошибка выжившего** | Копируем успешных, не видя кладбища неудачников | Изучай провалы, а не только успехи |
| **Confirmation Bias** | Ищем цифры, которые нравятся нам | Проверяй гипотезы, которые хочется опровергнуть |
| **Проклятие знания** | Думаем, что юзеру всё понятно | Проводи UX-тесты с новичками |
| **HiPPO** | Мнение начальника = истина | Мнение — это гипотеза. Проверяй цифрами |
| **Simpson's Paradox** | Смешиваем когорты (Новые + Старые) | Всегда сегментируй данные |

### 2.2 Эффект Слона (Blind Men & Elephant)
В зоне **некомпетентности** команда склонна:
1.  Подтверждать свою точку зрения
2.  Выбирать факты, которые "удобны"
3.  Фильтровать слова и факты, которые не подтверждают их позицию

**Инсайт:** Каждый отдел (Маркетинг, Продажи, Дизайн, Разработка) "ощупывает слона руками" и видит только свой кусок. Никто не видит целого. Задача CPO — дать всем **общий контекст**.

### 2.3 Эхо-камеры
Разные отделы используют **одни и те же слова** с **разными значениями**:
*   "Конверсия" для маркетолога ≠ "Конверсия" для продакта
*   "Воронка продаж" для сейлза ≠ "Воронка продаж" для дизайнера

**Решение:** Единый глоссарий терминов + общий дашборд метрик.

## 3. Ловушки планирования

### 3.1 Наивный прогноз vs Реальность
```
План: "100 задач за 10 недель"
Реальность: Скорость падает, сложность растёт, технический долг накапливается
```

### 3.2 Эффект дрожащей руки
*   Добавление даже **5% задач** раздувает сроки на **50%** из-за переключений контекста.
*   Каждая "маленькая фича" тащит за собой хвост зависимостей.

### 3.3 Закон 90/90
> Первые 90% кода пишутся за 90% времени. Оставшиеся 10% — за вторые 90%.

**Вывод:** Закладывай буфер x2 на "последние 10%".

# ЧАСТЬ II: ЭКОНОМИКА ПРОДУКТА

## 4. Unit Economics & P&L

### 4.1 Водопад денег (P&L Structure)
```
Revenue (Выручка)
   └─ COGS (Себестоимость: серверы, API, смс)
      └─ Gross Profit (Валовая прибыль)
         └─ Acquisition Costs (CAC, Маркетинг)
            └─ Fixed Costs (Зарплаты, офис, софт)
               └─ EBITDA / Net Profit (Чистая прибыль)
```

### 4.2 Золотая формула (AMPPU)
```
Gross Profit = Users × (-CPUser + C1 × AMPPU)
```

| Метрика | Расшифровка | Что улучшать |
|---------|-------------|--------------|
| **CPUser** | Cost Per User (привлечение) | Каналы, креативы |
| **C1** | Конверсия в первую покупку | Онбординг, Activation |
| **AMPPU** | Average Margin Per Paying User | Ценообразование, Retention |

### 4.3 Железные правила
1.  **CPA < LTV** — Стоимость привлечения платящего клиента ВСЕГДА меньше его lifetime value.
2.  **Считаем в рублях** — Не "+0.5% конверсии", а "+3 млн рублей выручки".
3.  **Сравнивай с Burn Rate** — Эффект фичи должен быть больше, чем зарплата команды за время её разработки.

### 4.4 Финансовое Дерево Метрик (Complete Hierarchy)
Как **EBITDA** связана с **кликом в рекламе**:

```
Level 1: EBITDA ← Profit Net ← Profit (minus Fix Costs)
Level 2: Profit (Gross) = Revenue - Acquisition Cost (с НДС)
Level 3: Revenue = Buyers × Av.Price × Payment Count (LTV)
Level 4: Acq Cost = Buyers × CAC
Level 5: Buyer = Leads × Conv(Lead→Buyer)
Level 6: Lead = Users × Conv(User→Lead)
Level 7: User = Clicks × Conv(Click→User)
Level 8: Clicks = Impressions × CTR
```

**Связующая метрика:** `AMPPU = Av.Price × Margin`

**Смысл:** Любое изменение внизу (например, CTR баннера +1%) по цепочке бьет в EBITDA. Продакт должен видеть **всю цепь** и находить **рычаг** с максимальным мультипликатором.

## 5. Когортный анализ (Layer Cake)

### 5.1 Что такое когорта
**Когорта** — группа людей, пришедших в одном периоде ("одноклассники по месяцу регистрации").

### 5.2 Слоёный пирог
```
Выручка сегодня = "Хвосты" старых когорт + Новая когорта
```

### 5.3 Retention — ключ к устойчивости
| Паттерн Retention | Диагноз | Прогноз |
|-------------------|---------|---------|
| Падает в 0% | "Дырявое ведро" | Бизнес умрёт без постоянного притока новых |
| Выходит на плато | Устойчивый бизнес | MAU будет расти органически |
| Растёт | Product-Market Fit | Можно масштабировать |

### 5.4 Данные для когорт (Raw Data)
**Минимальный набор полей:**
```
User_ID | Date_Time | Order_ID | Transaction_ID | Status | Item | Amount | Margin
```
*Без сырых данных вы не сможете посчитать Retention и LTV правильно.*

---

## 6. Дерево метрик (Driver Tree)

### 6.1 Связь маркетинга и денег
```
Impressions → CTR → Clicks → CPC → CPUser → Conv(C1) → Buyers → CAC
```

### 6.2 Магия знаменателя
*   Увеличение конверсии (C1) всего на **+0.3%** может снизить CAC **в 2 раза**.
*   Это даёт **кратный рост** прибыли без увеличения расходов на трафик.

**Вывод:** Оптимизация конверсии — самый дешёвый рычаг роста.

# ЧАСТЬ III: АНАЛИТИКА И ПРОЦЕССЫ

## 7. Модель "Минное поле" (Conversion Funnel)

### 7.1 Воронка — это не труба, это минное поле
| Шаг | Этап | Выжило (пример) |
|-----|------|-----------------|
| 0 | Каналы (Impressions) | 100% |
| 1 | Лендинг | 80% |
| 2 | Корзина | 30% |
| 3 | Регистрация | 10% |
| 4 | Оплата | 0.4% |

### 7.2 Ключевой инсайт
**Дизайн первой сессии критичен.** Он должен мгновенно активировать пользователя в JTBD-сценарий. Каждый лишний клик — это мина, на которой "взрываются" пользователи.

---

## 8. HADI-циклы и гипотезы

### 8.1 Экономика гипотез
*   **Правило 1/10:** Успешна только 1 гипотеза из 10.
*   **Вывод:** Проверяй гипотезы **дешево и быстро**, чтобы найти "единорога".

### 8.2 HADI Framework
```
H — Hypothesis (Гипотеза)
A — Action (Действие / Эксперимент)
D — Data (Данные / Результат)
I — Insight (Вывод / Следующая гипотеза)
```

### 8.3 Fail Fast
*   Убивай нерабочие идеи быстро.
*   Каждая неделя с "зомби-фичей" — это Cost of Delay.

---

## 9. Cost of Delay (Цена задержки)

### 9.1 Что такое Cost of Delay
Задержка релиза на 2 месяца — это **не просто** выплата зарплат. Это:
1.  **Недополученная прибыль** за 2 месяца
2.  **Упущенные возможности** реинвестирования
3.  **Рост конкурентного давления**

### 9.2 Time to Market — Король
> Лучше кривой релиз сегодня, чем идеальный никогда.

Рынок не ждёт. Первый, кто занял нишу, получает преимущество.

## 10. Приоритизация (RICE)

| Фактор | Описание | Оценка |
|--------|----------|--------|
| **R**each | Сколько пользователей затронет? | Число |
| **I**mpact | Влияние на ключевую метрику? | 0.25 / 0.5 / 1 / 2 / 3 |
| **C**onfidence | Насколько уверены в оценках? | % |
| **E**ffort | Трудозатраты (человеко-недели) | Число |

**Формула:** `RICE = (Reach × Impact × Confidence) / Effort`

# ЧАСТЬ IV: СТРАТЕГИЯ И РОСТ

## 11. 16 ключей к продуктовой стратегии

### Поиск сегмента
1.  **Боль:** Ищи сегмент, у которого болит **сильнее всего**.
2.  **Важность × Частота:** Решаем задачу, которая критически важна и возникает часто.
3.  **Лидеры мнений:** Запускайся на тех, кто влияет на остальных.
4.  **Избегай гигантов:** Не лезь в контекст, где уже сидит большой игрок.
5.  **Быстрая петля:** Ищи сегмент, где цикл обратной связи максимально короткий.

### Позиционирование
6.  **Последовательность:** Сначала реши предыдущую задачу в цепочке действий клиента.
7.  **Уровень:** Решай задачу более высокого уровня (стратегическую).
8.  **Фича vs Продукт:** +1 соседняя работа = новая фича.
9.  **Каннибализация:** Убей низкоуровневую работу, если она мешает.

### Рост
10. **Виральность:** Ищи работы, которые по своей природе цикличны или виральны.
11. **Pivot:** Сместись в Low-cost или Premium, если в середине тесно.
12. **Тренд:** Найди маленький сегмент, который оседлал растущий тренд.
13. **B2C ↔ B2B:** Иногда решение для физлиц отлично заходит бизнесу.
14. **10X:** Придумай решение, которое **в 10 раз лучше** текущего стандарта.

## 12. Петли роста (Growth Loops)

### 12.1 Virality Loop
```
Пользователь → Получил ценность → Пригласил друга → Друг стал пользователем → ...
```
**K-factor = invitation rate × acceptance rate**
*   K > 1 — органический рост
*   K < 1 — нужен платный трафик

### 12.2 Retention Loop
```
Пользователь → Получил ценность → Вернулся за добавкой → Получил ещё больше ценности → ...
```

### 12.3 Activation
**Мост** между User Acquisition и Sales.
*   Без активации нет продаж.
*   Цель: активировать **80%** пользователей в **2-3 частотных JTBD-сценария**.

---

## 13. 9 способов масштабирования

Когда экономика сходится (**LTV > 3×CPA**), включаем форсаж:

| # | Канал | Когда использовать |
|---|-------|-------------------|
| 1 | **Paid Ads** | 3×CPA < AMPU |
| 2 | **Viral & Referral** | Есть сетевой эффект |
| 3 | **Organic: SEO/ASO** | Долгосрочная игра |
| 4 | **Direct Sales** | High-ticket, B2B |
| 5 | **Telemarketing** | Высокий чек, низкий volume |
| 6 | **Партнёрки (CPA сети)** | Масштабирование без риска |
| 7 | **SMM / Друзья** | Ранняя стадия |
| 8 | **Конференции** | B2B networking |
| 9 | **Личный бренд / Storytelling** | Thought leadership |

# ЧАСТЬ V: ПСИХОЛОГИЯ И ДИЗАЙН

## 14. JTBD (Jobs To Be Done)

### 14.1 Анатомия сценария
```
Когда <ситуация/триггер>
Пользователь видит <что он видит на экране>
Пользователь хочет <мотивация>
И может/делает <действие>
Чтобы получить <выгоду>
А мы хотим <бизнес-цель>
И мы делаем <как влияем на пользователя>
```

### 14.2 Глубина активации (Activation Depth)
| Сценариев активировано | Риск | LTV |
|------------------------|------|-----|
| 1 | Высокий отток | Низкий |
| 2-3 | Средний | Средний |
| **5-7** | Низкий | **Высокий** |

**Цель:** Двигать юзера от "попробовал" до "использует регулярно 3+ сценария".

### 14.3 Pain of Paying
> Люди любят покупать, но **ненавидят платить деньги**.

**Решение:** Делай оплату незаметной (auto-renew, one-click, токены вместо денег).

## 15. Игровые механики (Retention Loops)

### 15.1 Дофаминовый цикл
```
1. Триггер (пуш, уведомление)
   ↓
2. Действие (убил монстра, решил задачу)
   ↓
3. Награда (золото, статус, прогресс)
   ↓
4. Инвестиция (потратил ресурс на прокачку)
   ↓
   → Возврат к триггеру
```

### 15.2 Payment Wall (Wait or Pay)
*   Сажаем на крючок через бесплатную ценность
*   Создаём **дефицит времени** (ждать 12 часов)
*   Монетизируем **нетерпение** (заплати, чтобы не ждать)

### 15.3 Психология виральности
*   **Эмоции:** "Душераздирающая история" или "Треш/Злость" расшариваются лучше радости.
*   **Цикл:** Value → Эмоция (Зависть/Негодование) → Share
*   **Инсайт:** Люди пишут, чтобы **не держать эмоцию в себе**.

# ЧАСТЬ VI: УПРАВЛЕНИЕ И ОРГАНИЗАЦИЯ

## 17. Организационные ловушки

### 17.1 Ловушка "Квадратика" (The Box Trap)
*   **Проблема:** Топ-менеджмент спускает задачи по функциональным колодцам (Маркетинг, Продажи, Дизайн, Разработка).
*   **"Двойная сплошная":** "Не выходи за пределы своего квадрата!"
*   **Результат:** Все оптимизируют локальную функцию, а Продукт разрывается.
*   **Решение:** Кросс-функциональные команды, объединённые **целью**, а не функцией.

### 17.2 Броуновское движение (Anti-Pattern)
Когда нет чёткой цели (G), команда превращается в "Ветеранов Броуновского движения":
*   Все бегут в разные стороны
*   Куча действий, нулевой вектор движения
*   CPO дёргает ручки, запрещает, но не даёт цель

### 17.3 Лестница целей (Goal Ladder)
| Уровень | Что видит | Метрики |
|---------|-----------|---------|
| **CEO/Board** | Деньги | Revenue $1M+, EBITDA |
| **CPO/PM** | Сходимость экономики | AMPU > CPUser, Retention > 10% |
| **Team** | Релизы | MVP, выкатили фичу |

**Проблема:** Разрыв контекста.
**Задача продакта:** "Переводить" метрики команды в деньги бизнеса.

## 18. Принципы реализации (Delivery)

### 18.1 FFF (Fixed Time, Fixed Budget, Flex Scope)
*   **Срок и бюджет** — железобетонные
*   **Скоуп (функционал)** — гибкий
*   Если не успеваем — **режем функционал**, не двигаем дедлайн

> Лучше выпустить "скейтборд" вовремя, чем "Мерседес" никогда.

### 18.2 Scrumban & WIP
*   Ограничивай количество задач в работе (Work In Progress)
*   Главная книга: "Цель" (Голдратт)
*   **Ищи узкие места** потока

### 18.3 RAT vs Waterfall
| Подход | Процесс | Риск |
|--------|---------|------|
| **Waterfall** | Concept → Dev → Test → Launch | 200 млн руб в трубу (кейс ЛК) |
| **CustDev/RAT** | Discovery → Validation → Creation → Pivot | Сначала проверяем риски |

## 19. Лидерство и качество

### 19.1 Качество — это ползунок
*   Нет понятия "качественно/некачественно"
*   Есть "**достаточно для проверки гипотезы**"

### 19.2 Ситуационное лидерство
*   Иногда лидер должен "затащить команду на гору" на своей энергии
*   Показать вид сверху
*   Поделиться опытом — дальше команда пойдёт сама

### 19.3 OKR vs KPI
| OKR | KPI |
|-----|-----|
| Много неопределённости | Стабильный процесс |
| Мы многое меняем | Нужно упорно работать |
| Для инноваций | Для операций |

### 19.4 Критерии успешной работы
1.  Мы работаем без херни — если процесс не работает, меняем его
2.  Если самоходность команды снизилась — это ваша зона ответственности
3.  Иногда нужно затащить на гору — потом команда станет самоходной

## 20. Цикл Героя (Hero PM Framework)

### 20.1 Путь Product Manager
```
1. Подписывается под цель
   → Берёт ответственность за бизнес-результат (Деньги)
   
2. Делает диагностику
   → Анализирует бизнес, метрики, аудиторию
   → Использует количественные (метрики) и качественные (CustDev) методы
   
3. Находит узкие места и точки роста
   → Фокусирует команду на 1 рычаге
   
4. Выдвигает гипотезы
   → Убивает 90% идей, которые дают ~1-10 млн руб расходов
   → Приоритезирует
   → Защищает перед стейкхолдерами
   
5. Ставит задачи и формирует бэклог
   → Релизы, эксперименты
   
6. Проверяет гипотезу / выпускает релиз
   
7. Измеряет результаты
   → Делает выводы
   
8. Достигает результаты
   → Команда и собственники видят результат в деньгах
   → (или нет 😅)
   
ПОВТОРЯЕМ: цикл повторяется 3-4 раза до достижения результата (1-6 месяцев норм)
```

### 20.2 Минимальный набор артефактов
```
Goal (через OKR)
    └── Key Results → Целевые метрики
        └── Roadmap & Sprintlog
            └── Гипотеза
                └── JTBD-сценарий
                    └── Релизы + N экспериментов
```

### 20.3 Roadmap = Список гипотез
*   Роадмап — это **не список фич** "которые надо запилить"
*   Это список вопросов: "Если мы сделаем Х, то метрика Y вырастет на Z?"

# ЧАСТЬ VII: ADVANCED TACTICS

## 21. B2B Sales Hacking

### 21.1 Кейс Maxifier
*   **Было:** Продавали сложную "Симуляционную модель" (UI, настройки, обучение)
*   **Цикл сделки:** 7+ месяцев
*   **Стало:** Стали продавать "Value Report" — отчёт, где видны деньги
*   **Результат:** Цикл сделки сократился до **2 недель**

### 21.2 Урок
> Не заставляй клиента **работать** в твоём софте. Продавай ему **готовый ответ** (Отчёт/Вывод), который он может показать боссу.

### 21.3 B2B Блокеры (Скрытые враги)
| Блокер | Их мотивация | Как обойти |
|--------|--------------|------------|
| **Закупки** | Отжать цену | Показать ROI |
| **Юристы** | Минимизировать риски | Шаблонный договор |
| **Безопасники** | Запретить всё новое | Сертификаты, compliance |
| **IT** | Не менять работающее | Интеграция без нагрузки |

## 22. Экосистема (Super App Strategy)

### 22.1 Логика Super App
*   Захватить **частотные** сценарии (Такси, Еда, Музыка)
*   Переливать трафик в **маржинальные** (Банк, Страховка)

### 22.2 Монополия через активацию
> Если ты придумал, как активировать **80% людей** в его сценарии — ты победил.

### 22.3 Формула супер-приложения
```
Сценариев очень смежных (низкий барьер входа)
    +
Монопольное положение на рынке на долгое время
    +
Активация в 80% людей
    =
Super App
```

## 23. Зона комфорта vs Точки роста

### 23.1 Проблема
*   Команда проверяет гипотезы там, где **умеет** (Зона Компетентности)
*   Точки кратного роста лежат там, где команда **некомпетентна** (Зона Страха)
*   Срабатывает **1 из 10** или **3 из 100** гипотез

### 23.2 Промах по рычагу
```
Если точка роста = Конверсия (C1) → нужно менять продукт
А команда умеет только = Лить трафик (Users)
→ Команда будет сжигать бюджет, игнорируя реальную проблему
```

### 23.3 Вывод
> Идите туда, где **страшно и непонятно**. Там деньги.

## 24. Коммуникация и конфликты

### 24.1 Valley of Disappointment
*   **Ожидания:** Рост будет линейным
*   **Реальность:** Сначала провал (обучение, ошибки), потом экспонента
*   **Риск:** Именно в "яме" менеджеры паникуют и генерируют инциденты

### 24.2 Gap → Конфликты
```
Команда прошла путь A → B
Цель (G) находится на $1M+ OKR
Gap (разрыв) = где рождаются конфликты, эмоции
```

### 24.3 Правило 10 к 1
*   Между "Что я думаю" → "Что они поняли" — **пропасть**
*   10 шансов к 1, что возникнут трудности в общении
*   **Решение:** Оверкоммуникация. Повторяй главные цели 10 раз разными словами.

## 25. 99.8% Невидимок

### 25.1 Проблема
*   Команды фокусируются на **0.2%** тех, кто купил (Survivor Bias)
*   Мы не изучаем **99.8%**, кто НЕ купил

### 25.2 Инсайт
> Деньги и инсайты лежат в отказе, а не в покупке.

### 25.3 Вопросы для анализа
1.  Какие сценарии "болят" больше всего?
2.  Какие сценарии перестали болеть? (после релизов)
3.  Какие сценарии приоритетно взять в работу?
4.  Что не делает Рик, что приходится делать нашей команде?

# ЗАКЛЮЧЕНИЕ

## Главные принципы (The Golden Rules)

1.  **Деньги > Процессы** — Измеряй успех в рублях, не в задачах
2.  **Гипотезы > Мнения** — HiPPO — это просто гипотеза
3.  **Скорость > Качество** — Time to Market важнее идеального кода
4.  **Retention > Acquisition** — Дешевле удержать, чем привлечь
5.  **Фокус > Распыление** — Один рычаг лучше, чем 10 задач
6.  **Data > Intuition** — Confirmation bias — твой враг
7.  **Activation > Features** — Без активации нет LTV
